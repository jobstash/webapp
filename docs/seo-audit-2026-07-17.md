# JobStash Search Console and Google Jobs audit

Date: 2026-07-17
Scope: canonical `https://jobstash.xyz` host only (`www` intentionally excluded)

This audit was rerun from scratch after unrestricted network access was enabled. It used fresh live HTTP responses, the Search Console API, the URL Inspection API, the middleware APIs, and the current source tree. No network result from the earlier sandboxed run was reused.

## Executive result

Google is indexing JobStash job pages and has shown them in Google Jobs. The historical claim that no job ever indexed is disproved by current first-party data:

- Last 16 months: 637 clicks and 53,202 impressions on job-like URLs.
- Google Jobs appearances: `JOB_DETAILS` produced 171 clicks / 11,056 impressions; `JOB_LISTING` produced 353 clicks / 8,512 impressions.
- In the fresh URL Inspection sample, 94 of 201 completed job inspections were “Submitted and indexed.”

The real problem is inconsistent eligibility and URL lifecycle signals:

1. Many live pages emit invalid or semantically wrong location markup.
2. Old and current slugs both return `200`, so Google often keeps the old slug despite a different canonical.
3. The sitemap sources contain dead jobs, nonexistent/thin pillars, and stale counts.
4. Job `<lastmod>` is the publication timestamp, not the last significant page/markup change.
5. Job lifecycle data is incomplete: very old jobs remain available without `validThrough`, and the site has no Indexing API update/removal workflow.

The frontend location and canonical-slug bugs are fixed in this worktree. The sitemap-source and lifecycle problems require middleware changes.

## Sitemap inventory and submission fix

Fetched at `2026-07-17T16:08:57Z`:

| Corrected sitemap |   URLs | Purpose            |
| ----------------- | -----: | ------------------ |
| `/sitemap1.xml`   |      1 | Static pages       |
| `/sitemap2.xml`   |  5,000 | Jobs, first chunk  |
| `/sitemap6.xml`   |    378 | Jobs, final chunk  |
| `/sitemap3.xml`   |  3,000 | Pillars, first     |
| `/sitemap4.xml`   |  3,000 | Pillars, second    |
| `/sitemap5.xml`   |  2,134 | Pillars, final     |
| Total             | 13,513 | 13,513 unique URLs |

`robots.txt` is valid, allows crawling, and references `/sitemap.xml`. The XML
index and all six child documents were reachable and passed the official
sitemaps.org XSD, but the child documents were originally served below
`/sitemaps/` while advertising root-level page URLs. Google scopes a sitemap to
its own directory and descendants, so those page URLs were not allowed for the
child sitemap locations even though the XML itself was valid.

The corrected child documents are direct root-level `.xml` routes. The root
index is now static and does not call middleware, removing a second failure mode
where a slow or unavailable upstream could make the submitted index return
`500`. The short-lived `/sitemaps/*` routes permanently redirect to the
corresponding root-level child file, but the index itself contains no redirects.

### Job sitemap defects

- The sitemap source contains 5,378 unique job IDs.
- The current public job list contains 4,406 IDs, all of which are in the sitemap.
- 972 sitemap IDs are not in the public list.
- Of those 972, the details API returns `200` for 882 and `404` for 90.
- Therefore at least 90 current job sitemap entries are definitively dead URLs (1.67% of the job sitemap).
- Of the 882 off-list but reachable jobs, 352 are over 90 days old and 137 are over one year old. Age alone does not prove expiry, but these require an explicit active/expired lifecycle check.

The source's `timestamp` is used as `<lastmod>`. In this data model it is the posting/publication date, so a schema, description, apply URL, or lifecycle change does not update the sitemap recrawl signal.

### Pillar sitemap defects

The middleware source returns 21,137 pillar rows. The frontend keeps rows whose reported `jobCount >= 3`, producing 8,134 sitemap URLs. Those counts do not match the page API's active jobs:

- Deterministic random sample: 100 current pillar sitemap URLs.
- 61 returned indexable `200` pages.
- 24 returned `200` with `noindex` because the live page had fewer than three jobs.
- 15 returned `404` because the live page API returned no data.

Examples:

| Pillar               | Sitemap `jobCount` | Live active jobs | Page result   |
| -------------------- | -----------------: | ---------------: | ------------- |
| `i-alphalab-capital` |                 18 |                2 | 200 + noindex |
| `i-asteroid-capital` |                  5 |                1 | 200 + noindex |
| `o-cosmos`           |                  3 |                0 | 200 + noindex |
| `i-antler`           |                  3 |                0 | 404           |

This wastes crawl capacity and generates “submitted URL marked noindex/not found” outcomes. The 2.35 MB upstream pillar response also exceeds Next.js's 2 MB data-cache item limit during a production build, so its intended one-hour fetch cache is not applied.

Google has not downloaded it yet, so its coverage counters are not available. The pending state was not itself a parser verdict, but the child-path scope issue above needed correction before Google processed the index.

after the root sitemap index finishes processing successfully.

## JobPosting markup

Fresh live sample: 350 job URLs, of which 345 returned `200` and five returned `404`.

- All 345 successful pages contained parseable `JobPosting` JSON-LD.
- 111/345 omitted `jobLocation` entirely.
- 104/345 emitted both `TELECOMMUTE` and `jobLocation`; the “physical” locations were actually remote applicant-country mappings.
- Every successful sample omitted `validThrough`; 240 omitted `baseSalary`.

For the 94 indexed jobs with a rich-results result:

- 87 passed.
- Seven failed.
- Every critical error was `Missing field "jobLocation"`.
- `validThrough`, `baseSalary`, and partial-address fields were warnings. They must not be fabricated: `validThrough` is only required when an expiry is known, and salary must be the employer-provided amount.

The static location mapping was the main cause. Across the 4,406 public jobs, exact-key mappings resolved only 2,958 (67.1%). The mapping corpus had no exact entry for 597 of 1,160 current unique location strings.

After the frontend fix:

- 4,136/4,406 public jobs (93.9%) resolve to at least one truthful country.
- Remote coverage improves from 1,189/1,491 (79.7%) to 1,440/1,491 (96.6%).
- Of the remaining 270 jobs, 169 have no source location and 101 have an ambiguous/unparseable location. Their ordinary pages remain `index, follow`, but invalid `JobPosting` markup is omitted until a country can be established.

This distinction matters: missing JobPosting location blocks Google Jobs rich-result eligibility; it does not itself block ordinary web indexing.

## Tangem reproduction

Upstream data for `OYJ9hV`:

```json
{
  "location": "REMOTE",
  "locationType": "REMOTE",
  "organization": "Tangem"
}
```

Before the fix, lookup was case-sensitive. The mapping contained `Remote`, not uppercase `REMOTE`, so the page silently omitted all remote fields and Google reported `Missing field "jobLocation"`.

Search Console also showed a canonical split:

- Old path `.../data-analyst-acquisition-tangem-ag/OYJ9hV`: indexed, but JobPosting rich result failed on `jobLocation`.
- Current path `.../data-analyst-acquisition-tangem/OYJ9hV`: unknown to Google.
- Both paths returned `200`; the old page declared the new path canonical, but Google selected the old path.
- The old path recorded one click, 22 impressions, and average position 4, proving the job page did index.

Verified local output after the fix:

- Current path: `200`.
- Old path: `308` to the current path.
- JSON-LD: `jobLocationType: "TELECOMMUTE"`.
- JSON-LD: applicant countries present.
- JSON-LD: no fake physical `jobLocation`.
- Stable `identifier` uses `OYJ9hV`.

## Implemented frontend fixes

1. Location lookup now uses exact mappings first, then normalized case/whitespace matching, reviewed plain-location fallback for remote prefixes, and conservative country/locality inference.
2. Raw `location` and `locationType` survive DTO transformation, so schema decisions no longer depend on display tags.
3. Fully remote jobs emit `TELECOMMUTE` plus `applicantLocationRequirements` and do not turn applicant countries into physical offices.
4. Hybrid jobs are not marked as fully remote.
5. Physical jobs emit the most specific real address available; only `addressCountry` is mandatory.
6. When no truthful country, employer, description, or apply URL exists, the page remains indexable but invalid JobPosting markup is omitted.
7. JobPosting now includes a stable job identifier.
8. Any noncanonical slug returns a permanent redirect to the current job path.
9. Static generation uses the same canonical href slug as the job model.
10. Sitemap children are direct root-level XML files, the root index has no
    upstream dependency, and the final advertised chunks absorb inventory
    growth without changing the index.

Validation completed:

- 145/145 tests pass.
- TypeScript passes.
- Changed-file lint passes with zero warnings/errors.
- Production Next.js build passes.
- Local production smoke test confirms Tangem's `200`, its old slug's `308`, and corrected JSON-LD.

## Required middleware follow-up

These cannot be made reliable solely in the frontend:

1. Make `/v2/search/sitemap/jobs` return only URLs whose details endpoint is indexable and live. Exclude the 90 current `404` IDs.
2. Return a canonical path (or immutable slug) generated from the same source as `/jobs/details`; list/sitemap organization names currently diverge from detail-page canonical names for a material sample of jobs.
3. Make pillar `jobCount` count current public/applyable jobs, not historical associations. Return only sitemap-eligible pillars or expose an `isIndexable` flag.
4. Add `updatedAt`/`modifiedAt` and use it for sitemap `<lastmod>`; do not substitute publication date or the current time.
5. Add explicit lifecycle/expiry fields. Remove expired jobs, return `404/410`, remove JobPosting markup, or provide a past `validThrough` as appropriate.
6. Produce structured work mode and applicant/physical country data at ingestion. Runtime string inference should remain a safety net, not the authoritative source.
7. Add a Google Indexing API publisher for `URL_UPDATED` and `URL_DELETED` notifications. Google specifically recommends the Indexing API for short-lived JobPosting pages while retaining a sitemap for overall coverage.

After deploying both frontend and middleware fixes, send `URL_UPDATED` for valid job pages, `URL_DELETED` for removed jobs, wait for the new sitemap index to process, and rerun a smaller URL Inspection validation sample.
